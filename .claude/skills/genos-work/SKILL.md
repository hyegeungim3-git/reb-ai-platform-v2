# GenOS 작업 오케스트레이터

어떤 모델이 세션을 이어받아도 같은 품질의 과정·결과물이 나오도록 하는 표준 루프. **판단이 필요한 곳은 원칙을, 판단이 필요 없는 곳은 스크립트를 쓴다.**

## Phase 0: 컨텍스트 확인 (모든 작업 공통)

1. `genos-app/CLAUDE.md`를 끝까지 읽는다 — 아키텍처 2층 구조, 작업 규칙, 현재 상태, 배포 표.
2. 작업이 v3 로드맵 항목이면 `docs/V3-ROADMAP.md`의 해당 항목 수용 기준을 확인한다.
3. 기존 산출물 확인: 같은 주제의 직전 커밋·문서가 있으면 읽고 승계한다(수치·명명·패턴). 부분 수정 요청이면 해당 부분만 건드린다.
4. 수용 기준을 1~3문장으로 확정한다. 불명확하면 추천안과 함께 사용자에게 질문.

## Phase 1: 작업 유형 판별과 실행 모드

| 유형 | 실행 | 비고 |
|------|------|------|
| 도메인 팩 콘텐츠 (신규 팩·심화·시나리오·XAI 데이터) | `pack-author` 에이전트 위임 (`model: "opus"`), 2개 이상 팩이면 병렬 스폰 | 세계관 제약(기존 수치)을 프롬프트에 명시해 전달 |
| 코어 일반화 (팩 필드 추가·컴포넌트 신설) | 메인 루프 직접 | 원칙: 코어에 콘텐츠 하드코딩 금지, REB fallback 유지, **코어 커밋과 팩 커밋 분리** |
| 검수 | `genos-verifier` 에이전트 위임 (general-purpose 타입, `model: "opus"`) 또는 소규모면 `genos-verify` 스킬 직접 실행 | 스크립트 우선, 수동 DOM은 변경 지점만 |
| 배포 | 메인 루프 직접 | 아래 Phase 4 |

실행 모드는 서브 에이전트(생성-검증 패턴)다: pack-author 산출 → genos-verifier 검증 → 실패 항목만 재작업. 이 환경에는 팀 도구가 없다.

## Phase 2: 구현

- 파일 수정은 Edit 도구만. PowerShell 치환은 한글 파일을 파괴한다.
- 새 팩 필드를 만들면 같은 커밋에서 `docs/DOMAIN-PACK-GUIDE.md` 스키마를 갱신한다 (스키마 표류 방지).
- 관리자 mocks.js에 새 상수를 추가하면 **export let·__REB_DEFAULTS·applyAdminDomain 3곳 등록** — 누락 시 조용히 무시된다.
- 함정을 밟기 쉬운 작업(빌드·자동화·배포·주석) 전에 `references/pitfalls.md`를 읽는다 — 전부 실제 사고 이력이다.

## Phase 3: 검증 (완료 = 빌드 + 실행 증거)

1. **자동**: `node .claude/skills/genos-verify/scripts/verify.mjs` — 3도메인 금칙어·핵심 마커·콘솔 에러. 결과 표를 보고에 포함.
2. **수동**: 변경 지점의 실제 조작 (버튼 클릭→상태 변화, 왕복 저장, 시나리오 완주). 스크린샷보다 DOM 텍스트 판정.
3. **빌드**: ASCII 복사 빌드 EXIT 0 + "✓ built in Xs" (절차·함정: `docs/QUALITY-CHECKLIST.md` §A). "transformed"까지만 보고 통과 판정 금지.
4. 규모 있는 작업이면 전역 `/quality-gate` 기준의 자기검토 7문을 통과시킨다.

## Phase 4: 배포와 기록

1. 커밋: 한국어, 무엇을·왜, 쌍따옴표 금지(here-string `@'...'@`). 코어/팩 분리.
2. push → `gh run watch <id> --exit-status`. **Pages 일시 오류(deploy만 실패)가 상습 재발** — 실패 런 rerun 금지, `gh workflow run`으로 새 런.
3. 라이브 검증: index 200 + **변경이 든 청크에서** 마커 문자열 확인. lazy 컴포넌트(OrchestrationScenario 등)는 index가 아니라 자기 청크에 있다. 'App-' 정규식은 \b 없이 쓰면 UserApp에 오매칭.
4. `CLAUDE.md` 현재 상태에 한 줄 추가(커밋 해시·검증 증거 포함). 하네스 자체를 고쳤으면 CLAUDE.md 하네스 변경 이력도 갱신.

## 품질의 기준 — 왜 이 루프인가

이 프로젝트에서 좋은 결과물의 공통점은 세 가지였다:
1. **같은 세계관**: 새 콘텐츠가 기존 콘텐츠(회의록 안건, 지도 인사이트, 조회 이력)와 수치·일정으로 맞물릴 때 데모 설득력이 생긴다. 고립된 콘텐츠는 아무리 좋아도 소품이다.
2. **증거 기반 완료**: 모든 완료 보고는 "실행했더니 이랬다"였다. 추정 보고는 이 프로젝트에서 7주짜리 빌드 오판을 만든 적이 있다.
3. **코어-팩 분리**: 도메인 콘텐츠가 코어에 스며드는 순간 다음 도메인 추가 비용이 폭증한다. 지름길처럼 보여도 하드코딩하지 않는다.

## 테스트 시나리오

- **정상**: "한빛정밀 팩에 구매 발주 시나리오 추가해줘" → Phase 0 문서 확인 → pack-author 위임(세계관: 기존 협력사·자재코드 수치 전달) → verify.mjs + 시나리오 완주 DOM → ASCII 빌드 → 팩 커밋 → push·CI·라이브 마커 → CLAUDE.md 갱신.
- **에러**: verify.mjs가 manufacturing에서 금칙어 1건 검출 → 해당 파일 grep으로 위치 특정 → pack-author에 해당 키만 재작업 지시 → 재검증 통과 후에만 커밋.
