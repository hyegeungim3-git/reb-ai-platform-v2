# GenOS v3 로드맵

> v2(태그 `v2.0` · 브랜치 `v2`, 2026-07-12 동결)까지: 3도메인 팩 × 시나리오 2장 체제, XAI(신뢰도·근거·기여도·HITL), 알림 센터, 피드백 루프, 핸드오프·브리핑, 13종 에이전트, 관리자 53페이지.
> v3의 방향: **"잘 만든 데모"에서 "살아있는 플랫폼처럼 느껴지는 데모"로.** 각 항목은 독립 실행 가능하며, 실행 시 `genos-work` 스킬의 표준 루프를 따른다.

## 실행 규칙

- 한 항목 = 한 사이클(구현→검증→배포→기록). 항목을 쪼개서 하다 말지 않는다.
- 코어-팩 2층 구조는 v3에서도 불가침. 모든 신기능은 "팩 필드 공급 시에만 노출" 패턴을 따른다.
- 각 항목의 수용 기준(DoD)을 충족하는 실행 증거 없이 완료 처리하지 않는다.

## P1. 라이브 데이터 엔진 — 시간이 흐르는 데모 (최우선 추천)

**무엇**: 지금 모든 수치는 정지 사진이다. 팩 시계열(가동률·수위·신고 건수·진동 RMS)에 tick 엔진(전역 스토어, 배속 조절)을 달아 지도 히트맵·알림·브리핑·에이전트 피드가 데모 중 실시간으로 변하게 한다. 임계 초과 시 알림 센터에 새 알림이 "도착"한다.
**왜**: 발표 데모의 몰입감을 결정하는 건 "살아있음"이다. 기존 예지보전·재난 세계관(임계치·수위 상승 추세)이 엔진의 시나리오 각본이 된다.
**DoD**: ①배속 1×/10×/60× 컨트롤 ②한빛 도메인에서 RMS가 임계 돌파하는 순간 벨 배지 +1과 알림 항목 생성 ③10분 고배속 방치 후 수치가 바닥/천장 고착 없음(회복 항) ④탭 전환에도 상태 유지(전역 스토어) ⑤3도메인 각 1개 라이브 지표.

## P2. 도메인 팩 스튜디오 — 노코드 발주처 교체

**무엇**: 관리자에 '도메인 팩 스튜디오' 페이지 — 기존 팩을 복제해 조직명·브랜드 컬러·페르소나·제안 카드·시나리오 제목을 폼으로 편집하고, localStorage 커스텀 팩으로 즉시 미리보기 + JSON 내보내기(_template.js 규격).
**왜**: "도메인을 갈아끼울 수 있는 범용 플랫폼"이라는 프로젝트 본질을 영업 현장에서 3분 안에 증명하는 기능. 신규 발주처 미팅에서 그 자리에서 그들의 조직명이 뜬다.
**DoD**: ①reb 복제→조직명·컬러 변경→포털 스위처에 4번째 도메인 등장 ②사용자 포털 3탭이 커스텀 값으로 렌더 ③JSON 내보내기가 DOMAIN-PACK-GUIDE 스키마와 일치 ④새로고침 후 유지·삭제 가능.

## P3. 감사 추적(Audit Trail) — AI 기본법 대시보드에 실데이터

**무엇**: 사용자 포털의 모든 AI 상호작용(질의·에이전트 실행·시나리오 완주·HITL 통과·피드백)을 localStorage 이벤트 로그로 축적하고, 관리자 AI기본법 대시보드·품질 관리가 이 실데이터를 집계 표시(고영향 처리 건수, 인적 확인 수행률, 신뢰도 분포 실측).
**왜**: 피드백 루프에서 시작한 "사용자단→관리자단 실데이터 흐름"의 완성. AI 기본법 제34조(기록·감독) 서사가 mock이 아닌 실동작이 된다.
**DoD**: ①시나리오 1회 완주 후 관리자에서 해당 실행 기록(시각·스테이지·HITL 지점) 조회 ②XAI '왜 이 답변인가' 열람율 등 지표 1개 이상 실측 집계 ③SECURE 세션은 로그 제외(무저장 서사) ④로그 상한·초기화 버튼.

## P4. 시나리오 빌더 — 오케스트레이션을 UI로 조립

**무엇**: 관리자 에이전트 빌더 확장 — 13종 에이전트를 드래그로 배열해 새 릴레이 시나리오(스테이지·로그·산출물 문구)를 만들고 사용자 포털 허브에 즉시 노출(localStorage 오버레이).
**왜**: "요청 1건→N개 에이전트 릴레이"가 하드코딩이 아니라 조립 가능한 프레임임을 증명. P2와 함께 '플랫폼성'의 양대 증거.
**DoD**: ①3스테이지 신규 시나리오를 UI로 생성→허브 카드 3장째 노출→완주 ②agentId 오타 방지(선택식) ③팩 orchestration 스키마와 동일 구조로 내보내기.

## P5. 접근성·PWA 마감 ✅ (2026-07-12 스코프 조정 후 완료)

**무엇**: PWA(manifest·아이콘·서비스 워커 오프라인 셸), Lighthouse 접근성 90+, verify.mjs 375 뷰포트 판정 확장.
**DoD**: ①Lighthouse 접근성 90+ ②PWA 설치 가능(manifest+SW) + 오프라인 셸 캐시 ③verify.mjs 3도메인 × (데스크톱 + 375) 통과.
**스코프 조정 기록**: 당초 포함했던 ⓐ다크모드 전면 지원은 표면적(13개 에이전트 화면 전부 재스타일) 대비 실수요 불명(발표 데모는 라이트 고정, 다크 서사는 SECURE 탭이 담당)으로, ⓑ챗봇 다국어 UI는 번역 에이전트와 서사 중복으로 **백로그로 분리**. 필요해지는 시점(야간 관제 데모, 해외 발주처)에 독립 항목으로 실행한다.

## 우선순위 근거

P1이 첫 번째인 이유: 투자 대비 데모 임팩트가 가장 크고, 기존 세계관 수치가 그대로 각본이 되어 콘텐츠 신작 비용이 0에 가깝다. P2·P4는 영업(플랫폼성 증명), P3은 규제 대응 서사, P5는 납품 품질. 사용자 우선순위에 따라 순서 조정 가능 — 단 P1을 하면 P3의 로그 이벤트 소스가 풍부해지므로 P1→P3 순서가 유리하다.
