# 도메인 팩 콘텐츠 작성법

스키마 정본은 이 스킬이 아니라 **`docs/DOMAIN-PACK-GUIDE.md`(필드 전체)와 각 에이전트 파일 상단 `CONTENT_DEFAULTS`(shape)**다. 이 스킬은 "좋은 팩 콘텐츠"를 가르는 원칙만 담는다. 작성 전 두 정본을 반드시 읽어라.

## 세계관 설계 — 팩 품질의 80%

좋은 팩은 필드 목록 채우기가 아니라 **하나의 살아있는 조직**을 만드는 일이다.

1. **대표 사건을 먼저 정한다.** 그 도메인의 실업무 사건 1~2개(예: 진동 알람, 호우경보, 이상거래 적발)를 정하고, 모든 필드가 그 사건을 서로 다른 각도에서 비추게 한다: suggestions(질문) → sampleAnswers(답변) → notifications(알림 도착) → orchestration(자동 처리) → agentRouting(핸드오프) → 회의록·mapIntel insight(배경 복선).
2. **수치는 단일 원장.** 사건의 숫자(건수·측정값·날짜·문서번호)는 한 번 정하면 팩 전체에서 동일하게 인용한다. 새 필드를 쓰기 전에 팩 파일을 grep해 기존 수치를 승계하라. 요일까지 확인하라(예: 2026-03-22는 일요일).
3. **인과가 맞아야 한다.** "임계치를 3/20에 낮췄다 → RMS 4.2는 구 임계치(4.5)로는 못 잡았다 → 조기 감지 첫 사례"처럼, 수치들이 논리적으로 서로를 필요로 하게 설계하면 데모 중 어떤 질문에도 세계관이 무너지지 않는다.
4. **업무 유형 쌍**: 오케스트레이션 시나리오는 도메인마다 "서류 트리거형(스캔 입수→OCR→...)" + "데이터/이벤트 트리거형(알람→조회→분석→...)" 한 쌍을 갖춘다. 후자는 attachment를 생략한다.

## 실업무 리얼리즘 체크

- 실제 시스템명(MES·PLM·RTMS·NDMS·새올·R-ONE), 실제 법령·표준 조항, 그 조직의 문서번호 체계(HBP-부서-연도-NNN)를 사용한다.
- 담당자 페르소나(이름·직급·부서)는 팩의 user·결재선·회의록 발언자와 일치시킨다.
- 처리 기준·임계값에는 근거를 붙인다("조례 별표 2", "ISO 20816"). 근거 없는 임의 숫자는 현업 검토자가 바로 알아챈다.

## XAI 필드 작성 (confidence·xai·factors·review)

- confidence: 근거가 튼튼한 규정 답변 93~97, 추정이 섞인 진단·예측 85~92, 근거 부족은 75 미만(HITL 배지 의도 노출용).
- xai.reasoning: "무엇이 결정적 근거였나" + **불확실성 1개 이상**(외삽·신뢰구간·대안 가설). 확신만 있는 설명은 오히려 신뢰를 깎는다.
- xai.rejected: 기각 사유가 정보가 되게 — "관련 없음"이 아니라 "점검 항목만 있고 존별 온도 데이터가 없음"처럼.
- orchestration 분석 스테이지에 factors(합계 ≈100%), 판정·발행 스테이지에 review(누가 승인하는지 실명 직책).

## 완성 판정

- [ ] 금칙어 0 — 타 도메인 조직·용어, KOGAS. 판정은 소스 grep이 아니라 DOM 스캔(genos-verify)
- [ ] suggestions 4개 중 지도 질의 1 + 답변 있는 질의 3 (답변 없는 카드를 남기지 말 것 — generic 폴백은 데모 김빠짐)
- [ ] sampleAnswers 키워드가 기존 항목·mapIntel과 충돌하지 않는 순서 (구체 항목 먼저)
- [ ] 오케스트레이션 실행 완주 + 결과 문서번호가 팩 체계와 일치
- [ ] 신규 수치가 기존 세계관과 모순 없음 (grep 대조)
